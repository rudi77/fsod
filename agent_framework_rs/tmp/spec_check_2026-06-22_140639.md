## Findings

### P1 - cli: one-shot with empty task exits successfully
**Source:** `agent_framework/docs/spec/cli.md` -> Invariants ("One-shot ohne Aufgabe und ohne weiteren Modus endet mit Fehlercode statt leerem Lauf.")
**Verdict:** FAIL
**Evidence:** `src/bin/agentkit.rs`:79-81 — when `--print`/`-p` is given with no task, the code prints `"Keine Aufgabe übergeben."` to stderr but then `return Ok(())`, yielding process exit code **0**. The spec requires a non-zero error code for this case.
**Suggested action:** Return a non-zero exit code (e.g. `std::process::exit(2)`) when no task is supplied, or update the spec invariant if exit 0 is now intended.

### P2 - cli: NO_COLOR environment variable not honored
**Source:** `agent_framework/docs/spec/cli.md` -> Capabilities/Config ("… abgeschaltet, wenn Konsole oder `NO_COLOR` sie nicht zulassen.")
**Verdict:** FAIL
**Evidence:** `src/bin/agentkit.rs`:50 — `let color = !args.no_color && std::io::stdout().is_terminal() && enable_vt();`. There is no check for `std::env::var("NO_COLOR")`, so a user setting `NO_COLOR=1` on a TTY still gets ANSI color.
**Suggested action:** Add `&& std::env::var_os("NO_COLOR").is_none()` to the `color` predicate.

### P2 - memory: ShortTermMemory truncate default differs from spec (2000 → 16000)
**Source:** `agent_framework/docs/spec/memory.md` -> Config (`truncate(limit=2000)` default)
**Verdict:** FAIL (documented default differs)
**Evidence:** `src/memory.rs`:33-36 — `TRUNCATE_LIMIT = 16000`, with an inline comment stating the larger limit is intentional to keep large coding tool outputs intact. The spec still documents `2000`.
**Suggested action:** Update the spec's documented default to 16000 (or note the coding-agent rationale), since the deviation is intentional and behavior-correct.

### P2 - planning / events: plan is delivered as a rendered string, not a structured Plan object
**Source:** `agent_framework/docs/spec/planning.md` -> API (callback receives the Plan) and `events.md` -> PLAN event data ("the Plan object")
**Verdict:** FAIL (contract narrowed)
**Evidence:** `src/planning.rs`:25 the update callback is `Fn(&[Step])`; `src/events.rs`:36 `EventData::Plan(String)`; `src/app.rs`:81 publishes the **rendered string**. The documented capabilities ("display the plan", "react to plan updates") work, but a consumer cannot access the full `Plan` object (e.g. call `render()` / inspect steps richly) from the PLAN event payload.
**Suggested action:** Either carry structured plan data in `EventData::Plan` (e.g. the step list) and pass the `Plan`/`&[Step]` to the callback, or update both specs to state the PLAN payload/callback is a pre-rendered string by design.

## Info (idiom differences — NOT drift)

These were flagged by per-spec checks as API-surface gaps but are README-documented Rust idioms with the capability fully preserved. Listed for transparency, not action.

- **sub-agents:** the `add_subagent()` convenience fn exposes fewer params than the Python kwargs version (`max_steps`, `param_name`, `param_desc`, `parallel_tools`) and returns `()` not the registry. Rust has no keyword/default arguments; all of these are configurable via the `Subagent` builder (`src/subagents.rs`:56-68). Capability intact.
- **mcp:** no explicit `close()` method / context-manager and no `name` parameter on `connect()`. Rust uses RAII `Drop` for cleanup; `name` is unused even in the Python original. (`src/mcp.rs`)
- **llm:** spec's "no transformation of provider chunks" invariant is Python-specific (it passes raw dicts). Rust wraps SSE deltas in a provider-neutral `Chunk`/`Delta`/`ToolCallDelta` model (`src/llm.rs`:174-200), explicitly documented as a deliberate difference. Streaming text + tool-call deltas behave identically.
